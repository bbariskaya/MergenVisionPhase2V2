8. CANDIDATE SELECTION ALGORİTMASINI EXACT UYGULA
==================================================

Mevcut problemler:

- Qdrant yalnız `top_k=1` sorgulanıyor.
- Stale ilk candidate sonrası ikinci candidate denenemiyor.
- Threshold üstünde ama PostgreSQL-invalid candidate varsa:
  `max(c.score for c in candidates if c.score < threshold)`
  boş generator ile ValueError oluşturabiliyor.
- Malformed payload/UUID adapter’ı crash ettirebilir.
- NaN/Inf candidate açıkça ele alınmıyor.

Exact algoritma:

1. `match_threshold` finite ve `[0.0, 1.0]` olmalı.
2. `candidate_limit` integer ve `1..100` arasında olmalı.
3. Qdrant `top_k=candidate_limit` ile sorgulanmalı; default 10.
4. Candidate’ları raw score descending sırala.
5. Non-finite score candidate’ı skip et.
6. Malformed point ID/payload UUID candidate’ı skip et; bütün request’i crash ettirme.
7. Raw score threshold altındaysa match kabul etme.
8. Threshold üstündeki her candidate için PostgreSQL’den doğrula:
   - identity var.
   - sample var.
   - identity active.
   - sample state == active.
   - sample is_active == true.
   - sample.face_id == identity.face_id.
   - candidate.face_id == identity.face_id.
   - candidate.sample_id == sample.sample_id.
9. İlk valid candidate kabul edilir.
10. İlk candidate stale, ikinci valid ise ikinci candidate kullanılmalı.
11. Hiç valid candidate yoksa new identity akışına gir.
12. Stale threshold-high score, new identity confidence’ını yükseltmemeli.
13. Empty `max()` kullanma:
    `max(values, default=0.0)` veya explicit list.
14. Production control flow’da `assert process is not None` kullanma.
15. Missing process/sample/identity typed consistency error üretmeli ve process mümkünse failed yapılmalı.

Zorunlu failing-first test dosyası:

backend/tests/integration/lifecycle/test_candidate_validation.py

Testler:

- first stale 0.99, second valid 0.98 -> existing face accepted.
- all stale -> new_anonymous, crash yok.
- missing sample -> skip.
- missing identity -> skip.
- inactive identity -> skip.
- pending/failed/inactive sample -> skip.
- sample.face_id mismatch -> skip.
- malformed payload UUID -> skip.
- exact threshold boundary -> accepted.
- NaN/Inf score -> skip.
- stale high score only -> new identity confidence 0.0.
- hiçbir process processing kalmıyor.

Service algorithm testlerinde scripted fake VectorStore kullanılabilir; gerçek Qdrant adapter integration testleri ayrıca korunmalı.

9. COSINE SCORE VE CONFIDENCE SEMANTİĞİ
==================================================

Qdrant cosine raw score teorik olarak `[-1,1]`.

RecognitionResult.match_confidence mevcut contract gereği `[0,1]`.

Tek helper yaz:

def _to_match_confidence(raw_score: float) -> float:
    if not math.isfinite(raw_score):
        return 0.0

Kurallar:

- Threshold karşılaştırması raw finite cosine score ile yapılır.
- Threshold kararından önce score’u clamp etme.
- Persist edilen confidence `[0,1]` bounded olur.
- Negatif raw score match sayılmaz.
- Negatif rejected score ile new identity sonucu confidence 0.0 olur.
- NaN/Inf candidate reddedilir.

==================================================
11. CROSS-STORE FAILURE VE COMPENSATION
==================================================

Caught failures için exact ordering ve reverse compensation uygula.

--------------------------------------------------
11.1 New identity flow
--------------------------------------------------

Normal order:

1. PG process + pending identity/sample reserve.
2. MinIO deterministic upload + stat/checksum verify.
3. Qdrant idempotent upsert wait=True.
4. Final PG sample active + result + process completed.

MinIO upload failure:

- sample failed.
- process failed.
- bu operation’da yeni yaratılan identity inactive/deleted.
- recognition_result yok.
- Qdrant upsert çağrılmamış.
- error message sanitized.

Qdrant upsert failure:

- MinIO object reverse-order delete denenir.
- sample failed.
- process failed.
- yeni identity inactive.
- result yok.
- Qdrant usable active point yok.

Qdrant success, final PG transaction failure:

- Qdrant point delete/deactivate compensation.
- MinIO object delete compensation.
- yeni bağımsız UoW ile mümkünse:
  - sample failed.
  - process failed.
  - new identity inactive.
- completed/result kalmamalı.
- Compensation failure sessizce yutulmaz.
- Typed partial-failure error ve sanitized failure details üret.
- PASS/completed sonucu dönme.

--------------------------------------------------
11.2 Existing candidate flow
--------------------------------------------------

Recognition result/final process commit başarısızsa:

- Sahte RecognitionOutcome dönme.
- Process mümkünse ayrı UoW ile failed yapılmalı.
- External sample/vector silinmez; bunlar önceden mevcut identity’ye aittir.
- Raw exception client/domain error mesajına sızmaz.

13. GERÇEK OPTIMISTIC LOCKING
==================================================

Mevcut durum optimistic locking değildir:

- Domain version artırıyor.
- Repository row’u select edip unconditional overwrite ediyor.
- Concurrent stale writer engellenmiyor.
- Deactivate version artırmıyor.

Exact implementation:

Repository port:

async def update_with_expected_version(
    identity: FaceIdentity,
    expected_version: int,
) -> FaceIdentity: ...

SQL tek atomic conditional statement kullanmalı:

UPDATE face_identity
SET
  status = ...,
  is_active = ...,
  display_name = ...,
  identity_metadata = ...,
  version = version + 1,
  updated_at = ...,
  deleted_at = ...
WHERE face_id = :face_id
  AND version = :expected_version
RETURNING ...

`rowcount != 1` veya RETURNING row yoksa:

ConcurrentUpdateError

Kurallar:

- Python object version’ını update öncesi keyfi artırıp unconditional persist etme.
- Domain desired state’i üretirken expected_version korunmalı.
- Successful DB update returned version source-of-truth olmalı.
- Enrollment ve deactivation bu repository metodunu kullanmalı.
- Deactivation da version artırmalı.
- İki session aynı version’ı okursa:
  - İlk update success.
  - İkinci deterministic ConcurrentUpdateError.
  - İlk writer’ın data’sı korunur.

Failing-first gerçek PostgreSQL test:

backend/tests/integration/persistence/test_optimistic_locking.py

Ek test:

- concurrent enroll/delete.
