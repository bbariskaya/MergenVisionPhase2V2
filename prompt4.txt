2. PHASE 1: FOTOĞRAF TABANLI KİMLİK SİSTEMİ
==================================================

Phase 1’in kullanıcı akışı şöyledir:

1. Kullanıcı bir fotoğraf gönderir.
2. Sistem input’un gerçekten desteklenen ve decode edilebilir bir görüntü olduğunu doğrular.
3. Fotoğraftaki bütün yüzleri bulur.
4. Her yüzü bağımsız olarak tanımaya çalışır.
5. Tek API çağrısı için tek processId üretir.
6. Her yüz için bounding box ve identity sonucu döner.
7. Sonuçlar ve process geçmişi daha sonra sorgulanabilir.

Bir görüntüde üç yüz varsa:

- Üç ayrı process oluşturulmaz.
- Bir process oluşturulur.
- Üç recognition_result kaydı oluşur.
- Her yüzün sonucu bağımsız olabilir.

Örneğin aynı response içinde:

- Bir kişi known
- Bir kişi anonymous
- Bir kişi new_anonymous

olabilir.

No-face bir sistem hatası değildir.

Geçerli fakat yüz bulunmayan görüntü:

- HTTP success döner.
- process completed olur.
- faceCount=0 olur.
- faces=[] olur.

Boş, bozuk veya desteklenmeyen dosya ise structured error döner. Raw CUDA, TensorRT veya decoder exception’ı kullanıcıya gösterilmez.

==================================================
3. IDENTITY SEMANTİĞİ
==================================================

Üç recognition result status’u vardır:

- new_anonymous
- anonymous
- known

Bunları birbirine karıştırma.

new_anonymous:

Sistem bu yüzü daha önce tanımamıştır ve ilk defa kalıcı bir faceId oluşturmuştur.

Bu yalnız ilk recognition sonucunun snapshot status’udur.

Persistent face_identity state’i new_anonymous değildir. Oluşturulan identity PostgreSQL’de anonymous olarak tutulur.

anonymous:

Sistem bu yüzü daha önce görmüştür.

Aynı kalıcı faceId bulunmuştur fakat kullanıcı henüz bu kimliğe isim vermemiştir.

known:

Aynı faceId kullanıcı tarafından enroll edilmiştir. Sonuç name ve metadata içerebilir.

Beklenen lifecycle:

İlk istek:
new_anonymous
faceId = A

Aynı yüz ikinci kez:
anonymous
faceId = A

Kullanıcı enroll eder:
faceId = A korunur
identity known olur

Aynı yüz tekrar:
known
faceId = A
name/metadata döner

Enrollment yeni faceId oluşturmamalıdır.

Eski recognition result’ları daha sonra değiştirilmemelidir. İlk sonuç sonsuza kadar new_anonymous snapshot’ı olarak kalır. Enrollment geçmişi yeniden yazmaz.

==================================================
7. DYNAMIC BATCH’İN AMACI
==================================================

Current image API bir request’te bir fotoğraf işler.

Buna rağmen engine’leri yalnız batch=1’e kilitlemiyoruz.

RetinaFace profile:

- min=1
- opt=4
- max=8
- spatial size 640×640

GlintR100 profile:

- min=1
- opt=8
- max=32
- crop size 112×112

Şimdiki image API RetinaFace’e batch=1 verir.

Dynamic detector batch’in amacı gelecekte video pipeline’da birden fazla decoded frame’i aynı native core’a verebilmektir.

GlintR100 batch’in amacı bir veya birden fazla görüntüde bulunan bütün yüz crop’larını verimli biçimde embed etmektir.

32’den fazla crop varsa deterministic şekilde chunk edilir; yüzler sessizce atılmaz.

Batch çalışırken detection ile embedding association kaybolmamalıdır.

Her detection şunları korumalıdır:

- hangi source image/frame’den geldiği
- detection index
- bbox
- landmarks
- embedding

6.1 Canonical API contract
----------------------------------------------------------------------

Canonical prefix:

Canonical image endpoint’leri:

Public JSON field isimleri camelCase olacak.

Python internal source snake_case kalabilir. Pydantic alias generator ve explicit schemas kullan.

Enum değerleri şu biçimde kalabilir:

new_anonymous
anonymous
known

Eski unversioned `/faces` ve `/processes` endpoint’leri gerekiyorsa deprecated alias olarak korunabilir; OpenAPI canonical contract `/api/v1` olmalı.

Her HTTP request:

- requestId üretir
- X-Request-ID response header’ında döner
- body bulunan response/error envelope’da requestId döner

Business operation response’ları ayrıca processId döndürür.

Query endpoint’leri yeni bir business identity operation yaratmak zorunda değildir; fakat requestId benzersiz call correlation’dır ve response ilgili processId/jobId’leri içerir. Bu normalizasyonu REFERENCE_DECISION_LOG’a kaydet.

Tek error envelope:

{
  "requestId": "<uuid>",
  "error": {
    "code": "UNSUPPORTED_MEDIA_TYPE",
    "message": "Safe client-facing message",
    "retryable": false,
    "details": {}
  }
}

Raw CUDA, TensorRT, SQL, MinIO, Qdrant, filesystem path veya exception string client’a dönmeyecek.

----------------------------------------------------------------------
6.3 Image orchestration correctness
----------------------------------------------------------------------

ImageRecognitionService’in bütün akışını guarded lifecycle içine al:

1. process start
2. input validation
3. native engine acquisition/init
4. inference
5. detection validation
6. per-face resolution
7. storage writes
8. process completion

Bu aşamaların herhangi biri fail olursa process kalıcı `processing` durumda kalmayacak.

Multi-face istekte bir yüz tamamlandıktan sonra ikinci yüz fail olursa:

- process completed sayılmayacak
- process failed olacak
- partial persisted immutable results silinmeyecek
- failure event partialResultCount içerebilir
- client’a sahte success dönülmeyecek

Failure persistence ayrı UoW ile best-effort ve sanitized code kullanmalı.

Compensation adımları birbirinden bağımsız çalışmalı:

- Qdrant cleanup fail olsa bile MinIO cleanup denensin
- MinIO cleanup fail olsa bile PG failure state/event yazılsın
- her cleanup sonucu ayrı kaydedilsin
- retry/reconciliation outbox’a bırakılabilsin

6.7 Native safety
----------------------------------------------------------------------

Mevcut native image runtime’ta:

1. pybind inference boyunca GIL’i bırak:
   - py::gil_scoped_release veya equivalent
   - Python object oluştururken GIL’i tekrar al

2. Execution slot için RAII lease oluştur:
   - normal return
   - C++ exception
   - CUDA error
   - Python cancellation
   durumlarının hepsinde slot release edilmeli.

3. Slot initialization state açık olsun:
   - initialized
   - unavailable
   - in_use
   ayrı kavramlar
   - constructor failure sessiz başarı sayılmasın

4. RetinaFace postprocess içindeki abort() kaldır:
   - structured native error döndür
   - bütün backend process’ini öldürme

5. Alignment status buffer’ını L2 status ile aynı buffer’da yanlış resetleme:
   - degenerate similarity transform detection rejected olmalı
   - hiçbir embedding üretilmemeli/persist edilmemeli

6. Model profile JSON gerçekten parse edilsin ve runtime tarafından kullanılsın:
   - tensor names
   - tensor dtypes
   - tensor shapes
   - input layout/color order
   - normalization
   - anchors/strides/variances
   - thresholds
   - landmark order
   - alignment template
   - model SHA
   - preprocess version
   - engine profile

7. Anchor/output mismatch:
   - no-face success değildir
   - MODEL_CONTRACT_ERROR olur

8. Exact profiles:

RetinaFace:
min = 1×3×640×640
opt = 4×3×640×640
max = 8×3×640×640

GlintR100:
min = 1×3×112×112
opt = 8×3×112×112
max = 32×3×112×112

32’den fazla recognition crop varsa detection order korunarak deterministic chunk yapılmalı; bütün request fail edilmemeli ve yüz atılmamalı.

9. CMAKE_CUDA_ARCHITECTURES hardcoded yalnız 75 olmasın.
   - build argument/config ile gerçek target architecture belirle
   - manifest’e kaydet

10. TensorRT base image Dockerfile’da tam digest ile pinlenmeli:
    tag@sha256:<full-digest>

11. Fresh checkout engine build path ekle:
    - scripts/build_engines.py veya shell-safe equivalent
    - exact trtexec/build command
    - model SHA
    - engine SHA
    - TRT/CUDA version
    - GPU compute capability/UUID
    - profile
    - container digest
    - build timestamp
    içeren manifest

Model ve engine Git’e commit edilmez.

Idempotency-Key mutating upload endpoint’inde zorunlu olsun.

Upload flow:

1. requestId üret
2. config normalize et
3. process/video/idempotency reservation yap
4. upload body’yi bounded stream et
5. aynı anda:
   - size count
   - SHA-256
   - secure local temp spool veya MinIO staging
6. bütün video RAM’e alınmayacak
7. ffprobe subprocess’i shell=False/list argv ile çalıştır
8. ffprobe timeout uygula
9. extension/MIME’a güvenme
10. gerçek container, codec, duration, width, height, time base, rotation doğrula
11. configurable limit kontrolü:
    - max bytes
    - max duration
    - max width/height
    - allowed container
    - allowed codec
12. staging MinIO object stat/size/checksum doğrula
13. canonical object’e finalize et
14. canonical stat/checksum doğrula
15. aynı PG transaction’da:
    - video_asset ready
    - video_job pending
    - process details
    - outbox/event
16. yalnız bundan sonra 202 dön
17. staging cleanup idempotent olsun

Job hiçbir zaman canonical video doğrulanmadan queue’ya girmeyecek.

Bozuk/boş/unsupported video:

- video_asset rejected
- process failed
- job yaratılmayabilir
- staging cleanup outbox
- typed safe error

Direct multipart upload bu fazda mandatory.

Presigned multipart şimdi zorunlu değil. Daha sonra eklenirse aynı finalize service’i kullanmalı; farklı validation path oluşturma.

Diğer endpoint’ler:

======================================================================

Redis ekleme. İlk implementation PostgreSQL lease queue kullanacak.

Claim:

1. kısa transaction aç
2. pending + available_at <= now job seç
3. SELECT FOR UPDATE SKIP LOCKED
4. state=processing
5. lease_owner
6. lease_expires_at
7. heartbeat_at
8. attempt_count += 1
9. transaction commit
10. GPU/external work transaction dışında yapılır

Worker:

- heartbeat interval configurable
- lease duration configurable
- max attempts configurable
- concurrent job count bounded
- GPU worker ownership açık
- unbounded queue/RAM yok

Expired lease recovery:

- processing/cancelling job lease expired
- cancellation requested ise cleanup/cancel path
- attempt_count < max_attempts ise pending + delayed available_at
- değilse failed
- process state buna uygun güncellenir
- duplicate identity/sample/vector üretmez

Progress:

- monotonic
- stage bazlı
- fake kesinlik üretme

- process completed
- job completed

- batch/stage boundaries’de kontrol edilir
- native subprocess graceful stop alır

- process exit beklenir
- temporary resources temizlenir
- lease bırakılır

13. MILESTONE 6 — PYTHON TRACKING VE RECONCILIATION

4. motion/IoU gating
5. deterministic tie breaking
6. tentative/confirmed/lost/removed state
7. scene cut veya uzun loss sonrası yeni rawTrackletId

UUIDv5(jobId, "raw:{ordinal}")

trackId:

UUIDv5(jobId, "track:{canonicalOrdinal}")

Reconciliation:

- fragmented raw tracklet’leri canonical track’e bağlar
- simultaneous overlapping tracklet’ler cannot-link
- transitive merge cannot-link’i ihlal edemez
- aynı frame’de iki ayrı yüz tek canonical track olamaz
- merge kararları deterministic ve explainable olur

Track template:

1. recognitionEligible observations
2. temporal diversity
3. quality’ye göre top candidates
4. cosine medoid/outlier rejection
5. quality-weighted average
6. tekrar L2 normalize
7. top1/top2/margin evidence

Embedding L2 normunu quality score sanma.

Threshold’lar config/model profile üzerinden gelir. Friends veya tek fixture’a göre source içine hardcode etme.

Identity resolution:

1. canonical track template ile Qdrant top-k
2. active=true + exact model_version filter
3. Qdrant candidate’ı PG’de active identity/sample olarak doğrula
4. top1 threshold
5. top1-top2 margin
6. candidate geçerliyse mevcut faceId
7. geçmiyorsa new anonymous creation path

Concurrent same-unknown protection:

- new identity yaratmadan hemen önce second Qdrant search
- stable non-reversible template bucket hash ile bounded PostgreSQL advisory transaction lock
- lock içinde tekrar search
- deterministic track/sample IDs
- idempotent Qdrant upsert
- duplicate creation integration testi

Embedding PG’ye yazılmayacak.

Bir canonical track için bir job içinde en fazla bir new faceId oluştur.

Aynı kişi farklı raw tracklet’lere bölündüyse reconciliation sonrası tek track/faceId olmalı.

İkinci videoda aynı anonymous kişinin aynı faceId ile bulunması mandatory acceptance case’tir.

  UUIDv5(trackId, "sample:{rank}")

- face_sample PG
