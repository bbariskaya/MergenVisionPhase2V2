Abi sorun büyük ihtimalle model değil: NV12 frame’i RGBA kernel’ine veriyor. Kanıt şu:

Frame: 1920x1080
Pitch: 2048
Gerçek RGBA pitch yaklaşık 1920×4 = 7680 civarında olmalı.
Kullanılan fonksiyon: mergenvision_preprocess_detector_rgba_batch
Decoder yüzeyi ise muhtemelen NV12; kernel Y/UV belleğini RGBA gibi okuyor.

Ayrıca ilk 16 frame’de gerçekten yüz olmayabilir. -88 değerleri “tensor sıfır” demek değil; normalize edilmiş koyu pixel olabilir. Threshold düşürmesin. Bunu hemen gönder:
ACİL SYSTEMATIC FIX — ZERO RETINAFACE DETECTIONS

Rastgele debug print, threshold değişikliği veya tracker çalışmasına devam etme. Önce detector input contract’ını kanıtla ve düzelt.

Mevcut kanıt:

  surface = 1920x1080
  pitch = 2048
  preprocessing function = mergenvision_preprocess_detector_rgba_batch
  detector output = 0 detections

1920 genişliğinde gerçek RGBA surface pitch yaklaşık width×4, yani en az 7680 byte civarında olmalıdır. 2048 pitch, surface’in RGBA olmadığını; büyük ihtimalle NV12 Y plane olduğunu güçlü biçimde gösteriyor.

Şu anda en olası root cause:

  NVDEC NV12 NvBufSurface
    → RGBA kabul eden CUDA kernel
    → Y/UV bytes yanlışlıkla RGBA channels gibi okunuyor
    → RetinaFace input tensor anlamsız
    → confidence düşük / 0 detection

`first10=-88` tek başına tensor doğru demek değildir. Border veya yanlış yorumlanmış luminance bytes olabilir.

Ayrıca Friends’in yalnız ilk 16 frame’inde yüz bulunmaması mümkündür. 16-frame test detector correctness fixture’ı değildir.

======================================================================
2. İKİ AŞAMALI ÇÖZÜM UYGULA
======================================================================

Aşama A — GPU RGBA correctness oracle

Önce root cause’u hızlı kanıtlamak için diagnostic/reference GPU path oluştur:

  nvv4l2decoder
    → nvvideoconvert
    → video/x-raw(memory:NVMM),format=RGBA
    → nvstreammux
    → appsink
    → mevcut RGBA CUDA preprocessing kernel
    → RetinaFace

`nvvideoconvert` CPU değildir; NVIDIA GPU conversion kullanır.

Properties:

  gpu-id = configured GPU
  nvbuf-memory-type = cuda-device
  compute-hw = GPU

Appsink tarafında doğrula:

  colorFormat == RGBA
  pitch >= width * 4
  memory is device-compatible

Bu path yalnız correctness oracle/debug baseline olarak kullanılacak.

Aşama B — production fused NV12 path

RGBA oracle yüz bulduktan sonra production hot path’i:

  NV12 NvBufSurface
    → fused CUDA NV12→model tensor resize/letterbox/normalize
    → RetinaFace

olarak implemente et.

Yeni kernel:

  mergenvision_preprocess_detector_nv12_batch(...)

NV12 kernel input’u frame başına açıkça taşımalı:

- Y plane device pointer
- UV plane device pointer
- Y pitch
- UV pitch
- source width/height
- plane offsets
- color matrix/range
- destination tensor pointer
- scale/padding metadata

Plane pointer’larını `base + planeParams.offset[n]` veya container’daki approved NvBufSurface API pattern’iyle oluştur. Offset/pitch’i tahmin etme.

NV12 conversion:

- negotiated caps/colorimetry’den BT.601 veya BT.709 seç
- limited/full range bilgisini doğrula
- 1080p diye körlemesine BT.709 hardcode etme
- Y ve interleaved UV’yi doğru plane/pitch ile oku
- RGB/BGR channel order model profile’dan gelsin
- pixel clamp ve normalization exact image-runtime contract ile aynı olsun

Fused kernel aynı launch içinde:

1. destination pixel → source coordinate mapping
2. bilinear NV12 sampling
3. YUV → RGB/BGR
4. letterbox fill
5. channel reorder
6. normalization
7. planar NCHW write

yapsın.

Ara RGBA full-frame allocation üretme.

8. FRAME LIMIT 304/300 DAVRANIŞINI DOĞRU DÜZELT
======================================================================

`--max-frames 300` demek:

  processed/inference/observation frame count == 300

demektir.

Mux son batch’i 296..303 olarak sekiz frame getirebilir. Bu durumda:

- 296,297,298,299 kabul edilir
- 300,301,302,303 `limit_discarded` olarak bırakılır
- inference’a gönderilmez
- observation yazılmaz
- tracker’a verilmez

Counters dürüst olsun:

- `decoder_emitted_count` 304 olabilir
- `accepted_frame_count` 300 olmalı
- `processed_frame_count` 300 olmalı
- `limit_discarded_count` 4 olabilir

`decoded_count == 300` diye sahte counter yazma.

Assembler, kalan kapasiteyi batch oluşturulmadan önce uygulasın:

  remaining = max_frames - accepted_count
  take = min(remaining, incoming_frames.size())

Kalan frame owner’ları güvenli release edilsin ve pipeline bounded biçimde durdurulsun.
