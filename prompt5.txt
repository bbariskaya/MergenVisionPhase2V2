A4. Fix video upload idempotency completely

Mevcut `_request_hash()` video bytes’ını içermiyor. Bu yanlış.

Şu davranış bağlayıcı:

same Idempotency-Key
+ same exact video bytes
+ same normalized sampling/model config
= aynı videoId/jobId/processId replay

same Idempotency-Key
+ different video bytes
veya different normalized config
= HTTP 409 IDEMPOTENCY_CONFLICT

Raw key yalnız `key_hash` üretmekte kullanılır. `request_hash` içine raw key koyma.

Yeni normalized request hash:

SHA256(
  videoContentSha256
  + canonicalSamplingMode
  + canonicalEveryNFrames
  + canonicalFramesPerSecond
  + modelVersion
  + preprocessVersion
)

Doğru sıra:

1. Upload’ı bounded temp spool’a stream et.
2. Aynı anda byte count + SHA-256 hesapla.
3. Her exit path’inde temp file finally ile silinebilir durumda olsun.
4. Normalized config üret.
5. request_hash hesapla.
6. PostgreSQL’de atomic idempotency claim yap.

Completed idempotency record bulunduğunda incoming body hash edilmeden cached response döndürme. Farklı bytes conflict’i ancak incoming SHA hesaplandıktan sonra anlaşılabilir.

Atomic claim:

- SELECT-then-INSERT yarışına bırakma
- PostgreSQL INSERT ... ON CONFLICT veya row lock kullan
- unique `(scope,key_hash)` contract’ını concurrency-safe kullan

Davranış:

- no existing row:
  atomik in_progress owner claim
- completed + same hash:
  cached IDs
- completed + different hash:
  409
- in_progress + same hash:
  409 IDEMPOTENCY_IN_PROGRESS, retryable=true
  veya bounded wait sonrası replay
- in_progress + different hash:
  409 IDEMPOTENCY_CONFLICT
- failed:
  failed snapshot expires_at’a kadar stable replay
  manual retry yeni idempotency key kullanır

Retry endpoint’inde de önce request_hash mismatch kontrolü yap, sonra cached completed snapshot döndür.
